## **Delivery & Deployment**

### **1. Delivery**

✦ Delivery means preparing the application and making it ready for release.

**Example:**

* Build application
* Run tests
* Create JAR/WAR file
* Store artifact

The application is ready, but users may not be using it yet.

---

### **2. Deployment**

✦ Deployment means moving the application to a server and making it available for users.

**Example:**

* Deploy WAR file to Apache Tomcat
* Deploy application to AWS EC2
* Deploy application to Kubernetes

Now users can access the application.

---

### **Simple Example**

Developer creates an application:

```text
Code
  ↓
Build
  ↓
WAR File Created
```

**This is Delivery.**

---

Then:

```text
WAR File
   ↓
Deploy to Tomcat Server
   ↓
Users Access Website
```

**This is Deployment.**

---

### **Easy Understanding**

```text
Delivery   = Ready to Release
Deployment = Running for Users
```

---

### **Real-Life Example**

```text
Delivery   = Food prepared and packed
Deployment = Food delivered to customer
```

---

### **Short Note**

✦ Delivery is the process of preparing the application for release.

✦ Deployment is the process of installing and running the application on a server for users to access.

## **Deploying Stages**

### **Steps Involved in Deployment**

1. Take a Java project from GitHub.
2. Clone the repository into the AWS Linux server.
3. Install Maven and JDK.

   * The JDK version should match the version specified by the developer.
4. Run the Maven Build Lifecycle.

   * The artifact is generated when the `mvn package` command is executed.
5. Install Apache Tomcat.
6. Copy the generated artifact (`.war` file) into the Tomcat `webapps` directory.
7. Start or restart Tomcat.
8. Access the application through a web browser.

---

### **Deployment Flow**

```text
GitHub Repository
        ↓
Clone Repository
        ↓
Install JDK & Maven
        ↓
Run mvn package
        ↓
WAR File Generated
        ↓
Install Tomcat
        ↓
Copy WAR File to webapps
        ↓
Start/Restart Tomcat
        ↓
Application Available to Users
```

---

### **Required Software for Deployment**

Before deploying a Java application, install:

* Git
* Maven
* Java (JDK)
* Apache Tomcat

---

### **Short Note**

✦ Deployment is the process of moving an application from the build stage to a server so that users can access it.

✦ Git is used to clone the source code, Maven is used to build the application, Java is required to run the application, and Tomcat is used to deploy the generated WAR file.

## **How to Deploy a Java Project**

### **Prerequisites**

Install the following software:

* Git
* Java (JDK)
* Maven
* Apache Tomcat

---

### **Clone the Java Project**

1. Open GitHub.
2. Copy the repository URL.
3. Clone the Java project into the server home directory.

```bash
git clone REPOSITORY_URL
```

---

### **Install Java and Maven**

#### **Update System Packages**

```bash
sudo dnf update -y
```

#### **Install Java 17**

```bash
sudo dnf install -y java-17-amazon-corretto-headless
```

#### **Install Maven**

```bash
sudo dnf install maven -y
```

---

### **Build the Java Project**

Go to the project directory:

```bash
cd FOLDER_NAME
```

#### **Validate**

```bash
mvn validate
```

#### **Compile**

```bash
mvn compile
```

Check the generated files:

```bash
ls
```

A `target` directory will be created.

Example output:

```text
target/classes
```

---

#### **Run Tests**

```bash
mvn test
```

Check the generated files:

```bash
ls
```

Example output:

```text
target/generated-test-sources
target/surefire-reports
```

---

#### **Package the Application**

```bash
mvn package
```

**Note:**

✦ If you directly run `mvn package`, Maven automatically executes:

```text
Validate
   ↓
Compile
   ↓
Test
   ↓
Package
```

✦ Running each stage separately helps verify whether every phase is successful.

✦ The `mvn package` command creates an artifact containing the complete application.

✦ The artifact is generated as a `.jar` or `.war` file inside the `target` directory.

---

#### **Verify**

```bash
mvn verify
```

#### **Install**

```bash
mvn install
```

---

### **Locate the Artifact**

Go to the target directory:

```bash
cd target
```

List files:

```bash
ls
```

Check the path:

```bash
pwd
```

---

### **Deploy the Artifact to Tomcat**

Copy the artifact to the Tomcat `webapps` directory:

```bash
cp ARTIFACT_NAME /opt/TOMCAT_NAME/webapps/
```

Example:

```bash
cp myapp.war /opt/apache-tomcat-9/webapps/
```

---

### **Verify File Permissions**

Go to the webapps directory:

```bash
cd /opt/TOMCAT_NAME/webapps/
```

Check permissions:

```bash
ls -l
```

Provide Read, Write, and Execute permissions:

```bash
sudo chmod 755 ARTIFACT_FILE_NAME
```

---

### **Start Tomcat**

Go to the Tomcat bin directory:

```bash
cd ..
cd bin
```

List files:

```bash
ls
```

Start Tomcat:

```bash
sh startup.sh
```

Expected output:

```text
Tomcat started successfully
```

---

### **Access the Application**

1. Ensure the Tomcat port is allowed in the Security Group.
2. Open a browser.
3. Enter the Server IP Address, Port Number, and Application Name.

**Example:**

```text
http://13.232.183.75:8080/FILE_NAME
```

or

```text
http://13.232.183.75:8080/myapp
```

---

        ↓

        ↓

### **Short Note**

✦ Maven is used to build and package the application.

✦ The generated artifact (`.jar` or `.war`) is stored inside the `target` directory.

✦ Tomcat deploys the artifact from the `webapps` directory and makes it accessible to users through a web browser.

---
### **Set Server Port Number**

1. Open the EC2 Instance.

2. Navigate to:

```text
Security → Security Groups → Edit Inbound Rules → Add Rule
```

3. Enter:

```text
Port Number : 8080
Source      : Anywhere-IPv4
```

4. Click **Save Rules**.

5. Copy the **Public IP Address** of the EC2 Instance.

6. Open a new browser tab.

7. Enter the Public IP Address along with the port number.

### **Default Tomcat Port**

```text
8080
```

### **Example**

```text
13.232.183.75:8080
```

or

```text
http://13.232.183.75:8080
```

### **Expected Result**

✦ If the Tomcat page appears, the server port is configured correctly and Tomcat is accessible from the browser.

### **Short Note**

✦ Tomcat uses port `8080` by default.

✦ The port must be allowed in the Security Group Inbound Rules before it can be accessed from a browser.
